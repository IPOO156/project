## 二、前端开发规范

### 2.0 前端开发核心原则

前端是用户直接接触和使用的部分，因此在开发过程中需要兼顾**易用性**与**美观性**。在页面设计和功能实现时，应以用户体验为核心，尽量简化操作流程，降低使用门槛。在满足业务需求的前提下，避免页面过于复杂和功能堆砌，优先考虑用户的使用便利性和操作效率，从而提升整体产品体验。

### 2.1 目录与命名规范

- **文件夹命名**：统一使用 `kebab-case`（如 `user-profile`）。
- **组件文件命名**：使用 `PascalCase`（如 `UserProfile.vue`）。
- **目录结构推荐（混合模式）**：

```
src/
├── features/          # 按业务模块划分（高内聚）
│   ├── student/       # 学生档案模块
│   ├── auth/          # 认证模块
│   └── ...
├── shared/            # 全局共享
│   ├── ui/            # 基础 UI 组件（Button/Input 等原子组件）
│   ├── utils/         # 工具函数
│   ├── api/           # 请求封装
│   ├── types/         # 全局类型
│   └── constants/     # 全局常量、字典
├── app/               # 应用入口（路由、全局样式、Provider）
├── assets/            # 静态资源
├── composables/       # 组合式函数
│   ├── usePagination.ts
│   ├── useTableQuery.ts
│   ├── usePermission.ts
│   ├── useDict.ts
│   ├── useUpload.ts
│   ├── useExport.ts
│   └── useFormValidate.ts
└── stores/            # Pinia 状态管理
```

- **路径别名**：在 `vite.config` 或 `tsconfig.json` 中配置 `@` 指向 `src`。
- **禁止跨层引用**；路径嵌套不要超过 4 层。
- **同一项目内禁止出现同名文件**（含不同目录）。

### 2.2 Vue 文件结构规范

推荐标准骨架（`<script setup>`）：

```vue
<script setup lang="ts">
// ── 1. 外部依赖导入 ──
import { ref, computed, watch, onMounted } from 'vue'

// ── 2. 类型/接口定义 ──
interface FormData { /* ... */ }

// ── 3. Props & Emits ──
const props = defineProps<{ /* ... */ }>()
const emit = defineEmits<{ /* ... */ }>()

// ── 4. Store / Composable ──
// const userStore = useUserStore()

// ── 5. 响应式数据 ──
const loading = ref(false)

// ── 6. Computed ──
// const isDisabled = computed(() => ...)

// ── 7. Watch ──
// watch(() => props.id, (val) => { ... })

// ── 8. 生命周期 ──
onMounted(() => { /* ... */ })

// ── 9. 方法函数 ──
async function handleSubmit() { /* ... */ }
</script>

<template>
  <!-- 模板内容 -->
</template>

<style scoped lang="scss">
/* 组件样式 */
</style>
```

- 模板中禁止出现复杂表达式，一律抽为 `computed`。
- `<style>` 必须添加 `scoped`；确需全局样式时使用 `:global()` 并注释说明。

### 2.3 样式规范

- **禁止大量内联样式**：仅允许用于动态计算的样式值，且需注释说明原因。
- **设计 Token 集中管理**：在全局样式文件或 CSS 变量中统一定义颜色、字体、间距、圆角、阴影等。
- **组件样式覆盖**：表格、按钮、弹窗、表单样式均依据设计 Token 统一；禁止在业务组件中直接覆写第三方组件库内部类名（应通过 `deep()` 限定作用域并在 CR 中说明理由）。

### 2.4 ESLint 与代码格式化

- **强制启用 ESLint**：项目初始化即配置，推荐 `@antfu/eslint-config` 或 `eslint-config-airbnb` + Vue 扩展。
- **禁止**任何形式的 `// eslint-disable` 注释（确需例外需提交 CR 由前端负责人批准）。
- **Prettier 配置**：
  - `printWidth`: 100
  - `singleQuote`: true
  - `semi`: false
  - `trailingComma`: 'all'
- **Git Hooks**：强制配置 `husky` + `lint-staged`，提交前自动执行 `eslint --fix` 和 `prettier --write`。
- **CI 门禁**：加入 `eslint --max-warnings 0` 检查，警告即阻断合并。

### 2.5 单文件代码量控制

- **硬性约束**：单文件行数 ≤ **600 行**（不含注释和空行可放宽至 650 行）。
- **拆分策略（按优先级）**：
  1. 弹窗类 → 抽取为独立组件，放置于 `./components/` 下。
  2. 查询表单 → 抽取为 `SearchForm.vue`。
  3. 复杂表格 → 抽取为 `DataTable.vue`。
  4. 重复的 template 片段 → 抽取为子组件或 slot。

### 2.6 组件化与复用

- **复用组件前置沟通**：预计被 ≥2 个页面/模块使用的组件，开发前必须在团队群同步（组件名称、Props 接口、使用场景）。
- **通用组件登记**：在 `docs/components.md` 中登记路径、用途、Props 说明、使用示例。
- **全局组件注册条件**（同时满足）：
  1. 在 ≥5 个不同页面/模块中使用
  2. 与具体业务逻辑解耦
  3. 仅通过 Props / Slots 与父组件交互

### 2.7 数据反显规范

- 字段映射统一在 API 响应拦截器或 Composable 中处理，禁止在模板中直接写复杂映射逻辑。
- 使用可选链 `?.` 和空值合并 `??` 处理空值/undefined。
- 日期/金额统一封装 `formatDate()`、`formatMoney()` 工具函数。
- 枚举/字典值使用 `useDict()` 统一转换，**禁止模板中硬编码**映射对象。

### 2.8 字典字段集中维护

- **集中位置**：`constants/dict.ts` 或 Pinia Store（`useDictStore`），二选一并在 README 中说明。
- **规范示例**：

```typescript
export const STUDENT_STATUS = {
  ACTIVE: { label: '在读', color: 'green' },
  GRADUATED: { label: '已毕业', color: 'blue' },
  SUSPENDED: { label: '休学', color: 'orange' },
} as const
```

- **禁止**在组件模板或 `<script>` 中直接定义字典映射对象；禁止在多个文件中重复定义同一字典。

### 2.9 状态管理与数据流

- **跨模块通信**（按推荐优先级）：
  1. 公共 Store（Pinia）— 全局共享数据
  2. 统一服务层 — 业务逻辑复用
  3. 事件总线（mitt / provide-inject）— 非父子组件通信（需注释说明事件流向）
  4. URL Query / Route Params — 页面间参数传递
- **禁止**直接修改另一模块的内部状态。
- **Store 设计**：按业务域拆分（如 `useStudentStore`、`useDictStore`），禁止单一巨型 Store；每个 Store 文件行数 ≤ 200 行。
- **数据流向**：修改代码前必须理清数据来源、去向及依赖关系。

### 2.10 页面布局规范

- 搭建主页面框架时必须规定并文档化：
  - 内容区最小/最大宽度（如 `min-width: 1200px; max-width: 1440px`）
  - 侧边栏宽度
  - 内容区最小高度
  - 表格/列表固定高度策略
- **响应式断点**：明确支持的屏幕尺寸范围（≥ 1366px 宽度为标准）。

### 2.11 模块间连接

- **页面之间**：通过路由参数和 Store 传递数据，禁止用 `localStorage`/`sessionStorage` 传递结构化数据（仅允许存储 token、用户偏好等）。
- **模块之间**：仅通过公共 Store / 统一服务层 / 事件总线通信，禁止模块间直接 `import` 对方的组件内部方法或响应式变量。
- **组件之间**：
  - 父子：Props down, Events up（单向数据流）
  - 兄弟：通过共同父组件或 Store 中转
  - 跨层级：`provide/inject`（需在 Provide 处注释说明下游消费者）

## 三、接口对接规范

### 3.1 前端对接原则

- 前端开发必须严格按照接口文档进行对接开发。后端在提供接口时，应提前明确接口定义和业务逻辑，无需等待接口实际开发完成后再进行沟通。
- 前后端应在开发前完成接口评审，确保双方对字段含义、业务规则及交互流程达成一致。
- 接口字段一旦确认，不得随意修改；如需调整，必须同步通知相关开发人员。
- 接口文档更新后，应及时同步给前端开发人员。
- 前后端对接口字段、业务逻辑存在疑问时，应先完成沟通确认后再进行开发。

### 3.2 接口文档必备内容（前端关注）

所有接口必须提供完整的接口文档，至少包含：

| 项目 | 说明 |
|------|------|
| 接口名称 | 接口功能描述 |
| 接口路径 | API 访问地址 |
| 请求方式 | GET、POST、PUT、DELETE 等 |
| 请求参数 | 参数名称及说明 |
| 参数类型 | String、Number、Boolean、Object、Array 等 |
| 是否必传 | 必填或非必填 |
| 参数说明 | 参数业务含义 |
| 返回参数 | 返回字段名称 |
| 返回参数类型 | 返回字段数据类型 |
| 返回参数说明 | 返回字段业务含义 |
| 错误码说明 | 常见错误码及对应含义 |

### 3.3 接口文档归类管理（前端需知）

接口文档统一存放路径：

```
D:\ruidao\laravel-first\NewRuiDao\docs\conclusion
```

接口文档分为两类，前端人员需了解以便对接和跟踪变更：

1. **总结接口文档（模块级全量文档）**：存放于 `\docs\conclusion`，用于记录某个模块的全部接口信息，属于系统级归档文档。
   - **注意：该文档生成时不能让 AI 修改，须统一手动新增或复制上去。**

2. **前端对接接口文档（交付文档）**：
   - **接口总对接文档（全量）**：用于前端完整对接某模块所有接口，强调完整性。
   - **接口修改文档（增量）**：用于记录接口变更内容，仅用于同步更新。
   - **注意：生成接口修改文档时，需要同步修改接口总文档，不能让 AI 修改。**

### 3.4 前端报错处理原则

为保证系统交互体验一致性，前端页面报错类型统一为以下几类：

1. **必填信息校验报错**：在对应输入框附近直接提示错误信息；用户修改后自动取消错误提示。
2. **接口频率限制报错（429）**：统一提示请求过于频繁；避免暴露系统内部技术信息。
3. **提示性报错**：明确说明原因，尽量引导用户下一步操作，避免使用技术术语。
4. **后端系统异常报错**：前端统一提示友好错误信息；不向用户暴露接口返回堆栈、SQL 信息等技术细节；提示用户联系管理员或稍后重试；**前后端均需对该类异常进行日志记录和监控告警**。

### 3.5 搜索功能验证

每个搜索字段必须验证：

- [ ] 单独搜索可用
- [ ] 组合搜索可用（至少测试 2-3 组常用组合）
- [ ] 重置功能可用
- [ ] 搜索值不清空时翻页，查询条件保持
- **防抖处理**：输入类搜索字段必须加 `300ms` 防抖。

### 3.6 Loading 状态规范

- **弹窗 Loading 必须覆盖以下场景**：

| 场景 | Loading 要求 |
|------|-------------|
| 弹窗打开 → 加载详情数据 | 整体 Loading（骨架屏/Spin 组件） |
| 弹窗内下拉框数据获取 | 下拉框自身 Loading 态 |
| 点击确定 → 调接口 | 确定按钮 Loading + 禁用态 |
| 弹窗关闭后再打开 | 清空旧数据 + 重新 Loading |

- **全局请求 Loading**：建议封装 Axios 拦截器统一处理请求 Loading 态。

### 3.7 字段来源规范

- **字段来源优先级（由高到低）**：
  1. 接口文档（Swagger / Apifox / YApi）
  2. 后端实际返回结果（发现与文档不一致需反馈后端修正文档）
  3. 数据库设计文档
  4. 产品原型标注
  5. 业务确认记录
- **红线**：禁止前端根据 UI 样式自行编造字段名；禁止将 Mock 字段名当作真实接口字段使用；禁止在未与后端确认的情况下修改字段名。
- **类型安全**：建议使用 TypeScript 定义接口返回类型，可通过 `openapi-typescript` 从 Swagger 文档自动生成类型定义。

### 3.8 数据变更后的更新

增/删/改操作完成后，必须保证：

- [ ] 当前列表/表单数据刷新
- [ ] 相关联的其他组件数据同步更新（通过 Store 或事件通知）
- [ ] 分页信息重置或保持合理状态

## 四、Git 使用规范

代码合并严格按照以下流程，**不可缺少，不可调整顺序**：

```bash
git add .
git commit -m "提交说明"
git pull origin 分支名
```

格式：**时间 + 方向 + 具体操作内容**

## 五、AI 使用规范

> **提示**：使用 AI 帮助修改开发代码时，必须传递本文件中的规范提示词。Claude Code 可直接读取本文件并将其内容保存在 memory 中（供后续会话持续参考）。

### 5.1 使用 AI 时必须提供的上下文

前端工程师在使用 AI 辅助开发时，Prompt 中必须明确提供：

| 类别 | 必填内容 | 示例 |
|------|---------|------|
| **项目信息** | 技术栈（Vue 3 + Pinia + Element Plus + TypeScript） | ✅ |
| **文件职责** | 当前修改文件的用途、在项目中的角色 | "该文件为学生列表页，负责展示、搜索、删除学生" |
| **已有上下文** | 相关 Store / API / Composable 的名称和用法 | "学生数据来自 `useStudentStore`" |
| **修改目标** | 具体要改什么、为什么改 | "在表格上方增加批量删除按钮" |
| **禁止改动范围** | 明确哪些文件/逻辑不要动 | "不要修改 `useStudentStore.ts` 中的 action 逻辑" |
| **约束条件** | 命名规范、字段来源、输出格式 | "字段名必须与 `api/student.ts` 中的接口返回一致" |

### 5.2 AI 使用红线（绝对禁止）

以下行为必须杜绝，违者 Code Review 直接打回：

| 禁止行为 | 说明 |
|---------|------|
| ❌ 大范围重构无关代码 | AI 生成代码时不应改动与需求无关的文件或逻辑 |
| ❌ 随意改变已有业务逻辑 | AI 不理解业务上下文，可能"优化"掉关键判断 |
| ❌ 删除注释中的重要业务说明 | 业务注释是团队知识的载体 |
| ❌ 引入未经确认的新依赖 | 每次引入新 npm 包需团队评审 |
| ❌ 伪造接口字段 | 字段必须来自接口文档或后端实际返回 |
| ❌ 把 Mock 数据当真实字段 | Mock 仅用于开发阶段，提交时需清理或显式标注 |
| ❌ 绕过类型检查 | 禁止使用 `as any`、`@ts-ignore`（确需例外需注释说明 + CR 审批） |

### 5.3 AI 生成代码后人工检查清单

每次 AI 生成代码后，开发者必须逐项确认：

- [ ] **是否改动无关文件？** — 使用 `git diff --stat` 检查变更范围
- [ ] **是否引入新依赖？** — 检查 `package.json` 是否新增依赖
- [ ] **字段是否对得上接口文档？** — 逐一对比 API 返回的字段名、类型
- [ ] **是否破坏原有样式？** — 运行项目，逐页对比改动前后的效果
- [ ] **是否存在重复逻辑？** — 搜索项目中是否已有相同功能的 Composable / 工具函数
- [ ] **是否能通过构建？** — 本地执行 `npm run build` 或 `pnpm build` 确认无编译错误
- [ ] **ESLint 是否通过？** — 执行 `npm run lint`

### 5.4 代码修改记录规范（前后端共用）

- 所有代码修改必须记录至指定地址。
- 文档命名规范：文件名格式 `YYYY-MM-DD.md`（自动获取当前日期）。
- 一次询问对应一次记录；用户的一次需求 = 一份修改记录，不得拆分多个文档。
- 修改记录内容：新增代码记录、删除代码记录；**必须先记录删除范围，再记录新增范围**。

## 七、版本管理与任务管理

### 7.1 版本号规范

采用语义化版本 `MAJOR.MINOR.PATCH`：

- `MAJOR`：重大功能上线或不兼容的 API 变更
- `MINOR`：新增向后兼容的功能模块
- `PATCH`：Bug 修复和小优化

### 7.2 版本控制要点

1. 具有两个主分支：服务端上线主分支、开发端开发用主分支。
2. 版本任务完成后，需在开发端主分支上对目标进行验收；验收完毕后才能上传到服务端，视为一次版本更新。
3. 每次部署到服务器时，在 Git 仓库打 `tag`（如 `v1.2.0`）。

### 7.3 任务跟踪

每项任务必须记录：任务名称、负责人、开始时间、完成时间、状态、关联需求、备注。

- **阶段性总结**：以双周为单位小总结，以月为单位大总结。
- **触发条件**：任务积压超过 10 项或延期超过 3 项时，必须发起专项总结会议。

## 九、关键提醒

1. **命名规律没有绝对对错**，只要写在项目根目录的 `CONTRIBUTING.md` 文档里，并用 ESLint 插件做约束，它就是好规律。
2. **发现代码风格冲突时**：暂停当前实现，主动找前端负责人确认；确认后写入 `docs/style-guide.md` 或更新 ESLint 规则。
3. **样式与布局问题必须立即反馈**（不可延至下个迭代）：页面无法自适应、表格无法滚动、弹窗层级错乱、文本溢出、交互区域过小等。
4. **后端部署通知**：后端部署时必须通过团队群通知前端，通知内容至少包含部署时间、部署环境、变更内容、影响范围、接口文档是否已同步更新。前端收到通知后 4 小时内完成对应页面的接口联调验证。
5. **新人入职**：须在 3 天内通读本文档并在任务管理工具中确认签字。
6. **每个迭代至少安排一次团队内技术分享**（30 分钟以内）。
