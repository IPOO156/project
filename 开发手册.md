# 二、开发规则

## 1.前端

**前端开发**：前端是用户直接接触和使用的部分，因此在开发过程中需要兼顾**易用性**与**美观性**。在页面设计和功能实现时，应以用户体验为核心，尽量简化操作流程，降低使用门槛。在满足业务需求的前提下，避免页面过于复杂和功能堆砌，优先考虑用户的使用便利性和操作效率，从而提升整体产品体验。

### 1.1页面报错处理原则

为保证系统交互体验一致性，前端页面报错类型应尽量标准化，统一为以下几类：

#### 1. 必填信息校验报错

适用于用户未填写必填项或输入内容不符合基础校验规则的场景。

**处理原则：**

- 在对应输入框附近直接提示错误信息。
- 明确指出缺失或错误内容。
- 用户修改后自动取消错误提示。

**示例：**

- 请输入用户名
- 手机号码不能为空
- 邮箱格式不正确

---

#### 2. 接口频率限制报错（429）

适用于用户请求过于频繁，被系统限流的场景。

**处理原则：**

- 统一提示请求过于频繁。
- 避免暴露系统内部技术信息。
- 可根据业务场景增加重试引导。

**示例：**

- 请求过于频繁，请稍后重试。
- 操作过于频繁，请稍后再试。

---

#### 3. 提示性报错

适用于业务规则限制或用户操作不满足条件的场景。

**处理原则：**

- 提示内容应明确说明原因。
- 尽量引导用户下一步操作。
- 避免使用技术术语。

**示例：**

- 当前数据已被删除，请刷新后重试。
- 您暂无该功能权限。
- 当前状态下无法执行此操作。

---

#### 4. 后端系统异常报错

适用于服务器异常、未知异常、数据库异常等非用户原因导致的错误。

**处理原则：**

- 前端统一提示友好错误信息。
- 不向用户暴露接口返回堆栈、SQL 信息等技术细节。
- 提示用户联系管理员或稍后重试。
- **前后端均需对该类异常进行日志记录和监控告警。**

**示例：**

- 系统异常，请联系管理员。
- 系统繁忙，请稍后重试。
- 服务异常，请联系管理员处理。

---

### 1.2前端对接原则

前端对接：前端开发必须严格按照接口文档进行对接开发。后端在提供接口时，应提前明确接口定义和业务逻辑，无需等待接口实际开发完成后再进行沟通。

前后端应在开发前完成接口评审，确保双方对字段含义、业务规则及交互流程达成一致。

#### 1.接口文档必备内容

所有接口必须提供完整的接口文档，至少包含以下内容：

| 项目         | 说明                                     |
| ------------ | ---------------------------------------- |
| 接口名称     | 接口功能描述                             |
| 接口路径     | API访问地址                              |
| 请求方式     | GET、POST、PUT、DELETE等                 |
| 请求参数     | 参数名称及说明                           |
| 参数类型     | String、Number、Boolean、Object、Array等 |
| 是否必传     | 必填或非必填                             |
| 参数说明     | 参数业务含义                             |
| 返回参数     | 返回字段名称                             |
| 返回参数类型 | 返回字段数据类型                         |
| 返回参数说明 | 返回字段业务含义                         |
| 错误码说明   | 常见错误码及对应含义                     |

#### 2.开发要求

1. 前端开发以接口文档为唯一对接依据。
2. 后端应在开发前完成接口字段设计和业务逻辑确认。
3. 接口字段一旦确认，不得随意修改；如需调整，必须同步通知相关开发人员。
4. 接口文档更新后，应及时同步给前端开发人员。
5. 前后端对接口字段、业务逻辑存在疑问时，应先完成沟通确认后再进行开发。

## 2.后端

#### 2.1开发前置要求

##### 1. 明确开发逻辑

后端开发开始前，必须充分理解并确认业务需求、业务流程及实现逻辑。

开发人员应明确：

- 功能目标及业务规则
- 数据流转逻辑
- 权限控制逻辑
- 异常处理逻辑
- 与其他模块的关联关系

禁止边开发边确认需求，避免因逻辑理解偏差导致重复开发。

##### 2. 先确定接口，再进行开发

开发前必须完成接口设计，并与前端进行确认。

接口设计应明确：

- 接口名称
- 接口路径
- 请求方式
- 请求参数
- 参数类型
- 是否必填
- 返回参数
- 返回数据结构
- 错误码定义
- 业务处理逻辑

接口定义确认后方可进入开发阶段。

禁止接口字段频繁变更或开发过程中随意修改接口结构。

#### 2.2接口变更管理

##### 1. 接口修改必须同步文档并通知前端

任何接口发生变更时，包括但不限于：

- 新增字段
- 删除字段
- 修改字段名称
- 修改字段类型
- 修改返回结构
- 修改业务逻辑

必须同步完成以下操作：

1. 更新接口文档；
2. 在变更记录中注明修改内容；
3. 主动通知相关前端开发人员；
4. 评估接口变更影响范围。

未经通知不得直接上线接口变更。

#### 2.3数据库变更管理

##### 1. 数据库修改必须公示

涉及数据库结构变更时，必须提前公示并记录。

包括但不限于：

- 新增数据表
- 删除数据表
- 新增字段
- 删除字段
- 修改字段类型
- 新增索引
- 删除索引
- 数据迁移

数据库变更需包含以下内容：

- 变更时间
- 变更人员
- 变更原因
- 影响范围
- 回滚方案

所有数据库变更脚本应纳入版本管理，确保可追溯、可回滚。

禁止直接在生产环境手工修改数据库结构而不留存记录。

**注：正式开发需要使用飞书来同步展示数据库修改，后端接口修改等，需要规定具体使用规范**

# 四、数据库设计

#### 1. 枚举优先，减少字符串存储

对于业务状态、类型、分类等固定取值字段，优先使用枚举值（Enum）或编码值，不直接存储展示文案， **尤其是需要后端进行对应使用的** 。

| 字段名          | 存储值            |
| --------------- | ----------------- |
| customer_status | CUSTOMER_STAY     |
| customer_status | CUSTOMER_UNDERWAY |
| customer_status | CUSTOMER_FINISH   |

对应含义：

| 枚举值            | 说明   |
| ----------------- | ------ |
| CUSTOMER_STAY     | 待审批 |
| CUSTOMER_UNDERWAY | 审批中 |
| CUSTOMER_FINISH   | 已审批 |

待审批/审批中/已审批

原因：

- 避免文案变更导致数据修改
- 方便后端逻辑判断
- 避免字符串拼写错误

---

#### 2. 设计表结构时优先考虑关联关系

建模时需明确实体之间的关系：

**2.1 一对一（1:1）**

例如：

```
用户(User)↓用户扩展信息(UserProfile)
user_profile------------iduser_id
```

---

**2.2 一对多（1:N）**

例如：

```
客户(Customer)↓联系人(Contact)
```

一个客户有多个联系人。

```
contact-------idcustomer_idname
```

---

**2.3 多对多（M:N）**

例如：

```
用户(User)↔角色(Role)
```

必须建立中间关联表：

```
user_role---------iduser_idrole_id
```

**不推荐**

```
user----role_ids = [1,2,3]
```

或

```
role_ids = "1,2,3"
```

原因：

- 无法建立外键
- 查询性能差
- 不符合数据库范式
- 难以维护

**设计原则**

一个字段尽量**不要保存数组、JSON数组、逗号分隔字符串**来表达关联关系。

---

#### 3. 关联关系统一保存 ID

关联字段统一保存主键 ID。

**命名规范**

格式：

```
表名_id
```

例如：

```
user_idcustomer_idrole_iddepartment_id
```

---

**多个相同类型关联时**

采用：

```
业务含义_表名_id
```

例如：

```
creator_user_id approver_user_id owner_user_id source_customer_id target_customer_id
```

避免：

```
user_id1user_id2
```

这种命名无法体现业务含义。

---

#### 4. 布尔值使用 BOOL/BOOLEAN

表示是/否、开启/关闭等状态时，使用布尔类型。

正确例：

```
is_deleted BOOLEANis_enable BOOLEANis_default BOOLEAN
```

---

不正确例：

```
is_deleted INT
0 = 否1 = 是
```

原因：

- 语义不明确
- 可读性差
- 容易出现非法值

例如：

```
is_enable = 2
```

理论上不应该存在。

---

#### 5. 补充建议

**主键统一**

```
id BIGINT
```

或

```
id UUID
```

全项目保持一致。

---

**时间字段统一**

正确：

```
created_atupdated_atdeleted_at
```

避免：

```
createTimecreate_datectime
```

多种风格混用。

---

**软删除统一**

推荐：

```
is_deleted BOOLEANdeleted_at DATETIME
```

不要直接物理删除业务数据。

---

**必要字段添加索引**

常见索引字段：

```
user_id customer_id status created_at
```

避免全表扫描。

---

**避免冗余字段**

如果数据可以通过关联查询获得，不要重复存储。

例如：

```
order-----user_id user_name   ← 不推荐
```

应通过：

```
user_id
```

关联用户表获取用户名。

# 五、后端接口文档规范

## 一、基本要求

后端开发在与前端对接过程中，**所有接口必须提供接口文档**，并严格遵守以下规则：

- 每个业务模块必须对应一份接口文档
- 接口新增、修改、删除后，必须同步更新接口文档
- 文档更新后必须及时通知前端
- 接口文档必须保持统一格式，禁止随意编写

---

## 二、接口文档字段规范

所有接口必须包含以下标准字段：

- 接口路径
- 请求方式
- 请求参数
- 请求参数类型
- 是否必须传入
- 返回参数
- 返回参数类型
- 备注

---

## 三、接口文档归类规范

接口文档统一存放路径：

```
D:\ruidao\laravel-first\NewRuiDao\docs\conclusion
```

接口文档分为两类：

---

## 5.1 总结接口文档（模块级全量文档）

**注意：该文档生成时不能ai修改统一手动新增或复制上去！**

**存放位置：**

```
\docs\conclusion
```

**定义：**

用于记录**某个模块的全部接口信息**，属于系统级归档文档。

**内容要求：**

必须包含该模块所有接口，并统一字段结构：

- 接口路径
- 请求方式
- 请求参数
- 请求参数类型
- 是否必须传入
- 返回参数
- 返回参数类型
- 备注

### 作用

- 模块级接口统一管理
- 便于系统整体维护
- 作为长期归档资料

---

## 5.2 前端对接接口文档（交付文档）

**ai生成存放位置：**

```
\docs\conclusion
```

用于前后端联调与版本同步，分为两个子类型：

---

### 5.2.1 接口总对接文档（全量）

**定义**

用于前端完整对接某模块所有接口。

**内容要求**

必须包含该模块**所有接口完整信息**：

- 接口路径
- 请求方式
- 请求参数
- 请求参数类型
- 是否必须传入
- 返回参数
- 返回参数类型

#### 特点

- 强调“完整性”
- 用于首次对接或整体联调
- 不包含变更说明

---

### 5.2.2 接口修改文档（增量）

**注意：生成该接口文档时需要同步修改接口总文档，不能让ai修改**

#### 定义

用于记录接口变更内容，仅用于同步更新。

#### 内容要求

必须包含：

- 接口路径
- 请求方式
- 请求参数
- 请求参数类型
- 是否必须传入
- 返回参数
- 返回参数类型
- 备注（必须说明变更内容）

#### 记录范围：

- 新增字段
- 删除字段
- 修改字段类型
- 返回结构变化
- 业务逻辑调整

#### 作用：

- 前端快速定位变更内容
- 降低阅读成本
- 支持版本迭代管理
